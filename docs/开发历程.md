# DeepBlue Rig 开发历程

> 海上油田数字孪生视觉模拟 · 人机协作开发全记录
>
> 本文按时间线还原整个项目从奠基到打磨的过程:**用户提出需求 / 反馈截图 → AI(Claude Code)实现并直推 GitHub `main` → Vercel 自动部署上线**。
> 每个阶段都附上对应的 Git 提交,可与代码一一对照。

---

## 一、项目概览

| 项 | 内容 |
|---|---|
| 目标 | 用游戏级渲染技术构建一座"活着的"海上油田数字孪生 |
| 技术栈 | React 18 · Vite 5 · TypeScript · Three.js (R3F 8 + drei) · @react-three/postprocessing · Zustand · Zod · Tailwind |
| 架构原则 | **渲染层 — 状态层 — 数据适配层** 三层解耦;场景只读状态、不读数据,为二期接入真实 SCADA 预留平滑通道 |
| 部署 | GitHub `main` → Vercel 自动构建上线(无需手动操作) |

### 协作方式

- **用户**:提出方向性需求("视觉精修""加海上活动")、给出真机截图反馈 bug、拍板技术方向。
- **AI**:研究代码 → 实现 → 本地构建验证 → 提交并直推 `main` → 说明改动与验证点。
- **闭环**:Vercel 部署上线后,用户在真机观察效果/发现问题,再进入下一轮。

## 三、深度复盘:夜间"炸白"bug 的诊断全过程

> 这段值得单独记录,因为它展示了一套"看不到现场也能定位 GPU bug"的方法论。

### 1. 现象

用户在真机上看到:夜晚场景里**月亮变成黑色圆盘(像日食)**,**海面在不同天气/视角下大面积变成黑白噪点或纯白**。而开发侧用无头浏览器(SwiftShader 软件渲染)截图,夜晚海面**始终正常**。

### 2. 逐层假设与排除

| 轮次 | 假设 | 结论 |
|---|---|---|
| 1 | 月亮黑盘 = 月亮球体未关 `fog`,被夜间近黑雾色染黑 | ✅ 成立,`fog={false}` 修复 |
| 2 | 黑白地平线 = 云层夜间不压暗,亮白云 vs 暗海 | ✅ 成立,加 `uDay` 压暗云层 |
| 3 | 海面纯白 = 着色器某项输出过大 | ❌ 按数学推导,夜间 `nightDim` 封顶 0.18,不可能白 |
| 4 | 海面纯白 = 逐像素 NaN | ✅ 方向正确,但兜底位置/方式有坑 |

### 3. 关键转折:把应用真正跑起来

读代码反复推导得出"海面不可能是白的",与真机现象矛盾——于是**停止盲猜,用 Playwright + 预装 Chromium 把应用跑起来截图**,在 22:00 各天气/各相机角度采样。结果:**软件渲染下夜晚海面完全正常**。

这个"软件渲染正常、真实 GPU 炸白"的**强不对称**,本身就是决定性线索:

> **NaN 在多数真实 GPU 上被渲染成纯白,而 SwiftShader 的 `sin` 做了范围归约、不会对超大参数返回 NaN,因此本地永远看不到。**

### 4. 根因

着色器里两类运算收到了**随 `uTime` 无限增大**的参数:

1. **噪声哈希** `fract(sin(dot(p, k)) * 43758.5453)` —— `p` 含 `uTime`,运行越久参数越大;
2. **Gerstner 波相位 / 粒子动画** `sin/cos(… + speed·k·uTime)` —— 同样无界增大。

真实 GPU 的 `sin` 在超大参数下精度崩溃乃至返回 NaN/Inf → 哈希/法线/顶点位移变 NaN → 被画成纯白。这也解释了为什么**跑一会儿、特定 GPU、特定天气(浪大/俯视)才炸**。

### 5. 为什么第一版"兜底"没用

`1202cc3` 在片元输出前用 `col.x == col.x ? col.x : fallback` 剔除 NaN。但用户截图显示仍炸白——推断 **GLSL 编译器把 `x == x` 折叠为恒真**,兜底失效。教训:**不要依赖运行时检测 NaN,要从源头消除产生 NaN 的运算**。

### 6. 根治方案

| 提交 | 手段 | 原理 |
|---|---|---|
| `4600bb1` | 4 处噪声哈希换成无 `sin` 的 **Hoskins 哈希** | 对任意大输入有界,不再 `sin(huge)` |
| `538c474` | 海面波相位 **对 2π 取模** | `sin/cos` 是 2π 周期,结果**完全等价、波形零变化、无跳变**,但参数恒在 `[0,2π)` |
| `b8e98fa` | 火炬烟雾/火星粒子的 `sin/cos(uTime)` 同样取模 | 同上,全工程清零隐患 |

至此,**全工程没有任何 GLSL 三角函数或噪声哈希会接收无界参数**,任意硬件、任意运行时长都不会再炸白。

### 7. 方法论沉淀

- **现象与代码矛盾时,先复现、再下结论**——别让推导骗了自己。
- **环境差异即线索**:软件渲染 vs 真实 GPU 的不一致,直接指向 NaN/精度问题。
- **从源头修,而非兜底**:运行时 NaN 检测可能被编译器优化掉。
- **利用周期性做无损改写**:`sin/cos` 取模 2π 是零代价、零副作用的稳健化手段。
